项目简介

RNOH (React Native for OpenHarmony) 是 OpenHarmony 系统级的 RN 渲染后端与原生桥接实现。开发者可使用 React 语法及 React Native 生态库，直接构建面向 HarmonyOS NEXT 的高质量应用。项目遵循 MIT 协议，深度融合 ArkUI，性能接近原生体验。

社区问题清单
整合 React Native 官方社区常见问题 & RNOH 鸿蒙适配专项问答
React Native 社区高频
  • [RN] 新架构 Fabric 如何与 RNOH 兼容？
    RNOH 完全支持 Fabric 渲染，需在 metro 配置中启用 newArchEnabled。
  • [RN] Hermes 在鸿蒙上性能表现？
    字节码预编译与内存优化已深度集成，性能接近 Android。
  • [RN] 三方库如 react-navigation 兼容性？
    大部分纯 JS 库可直接使用，涉及原生模块需桥接适配。
RNOH 鸿蒙生态专项
  • [RNOH] 签名失败 / 无法安装到真机
    需配置 DevEco 签名，华为开发者账号授权，参照文档配置 signingConfigs。
  • [RNOH] 升级到 0.82 后部分模块 undefined
    检查 TurboModule 注册方式变更，使用 Codegen 重新生成绑定代码。
  • [RNOH] 折叠屏适配最佳实践
    监听屏幕尺寸变化，使用 useWindowDimensions 动态布局，Modal 需启用分帧。

典型问题定位 · AI 辅助分析
RNOH 语料已进入 Gemini、DeepSeek 等主流 AI 模型
推荐使用 AI 工具（Copilot、通义灵码、DeepSeek Chat 等）结合日志与上下文快速定位问题。将错误堆栈 + RNOH 版本 + 设备型号提供给 AI，可获得根因分析与修复建议。
稳定性问题分析
📌 定位能力指导：
1. 收集崩溃日志：使用 DevEco Studio 获取 crash 日志或 tombstone。
2. 分析堆栈：重点关注 libRNOH.so、libreact_nativemodule.so 等符号。
3. 常见场景：线程未同步、NativeModule 生命周期错误、内存越界。
4. AI 提示词模板：“RNOH 0.82 在鸿蒙折叠屏上打开 Modal 闪退，堆栈如下：…… 请分析可能原因。”
5. 验证方案：隔离可疑模块，使用 AddressSanitizer 检测。
性能问题分析
📌 定位能力指导：
1. 抓取性能数据：React DevTools Profiler + 鸿蒙 DevEco Profiler。
2. 识别瓶颈：JS 线程阻塞、UI 线程掉帧、过度渲染。
3. 优化手段：FlashList 替代 FlatList、避免匿名函数、启用 Hermes 并发 GC。
4. AI 提示词模板：“RNOH 0.77 长列表滑动卡顿，JS 线程占用 80%，请给出优化策略。”
5. 验证：对比优化前后帧率曲线。
工程类问题分析
📌 定位能力指导：
1. 编译错误：检查 git submodule 是否更新、CMakeLists 依赖、RNOH 版本与 RN 核心版本对齐。
2. 链接错误：确认 so 库是否完整打包，使用 readelf 检查符号。
3. 运行时模块未找到：检查 TurboModule 注册与 Codegen 生成。
4. AI 提示词模板：“RNOH 编译报错 undefined symbol: jsi::Value，环境 0.82，如何解决？”
5. 复现与隔离：新建最小 demo 工程逐步添加模块定位。
推荐工作流：粘贴错误堆栈 + 版本信息 + 设备型号 → AI 提供根因与补丁建议，提效 80%
